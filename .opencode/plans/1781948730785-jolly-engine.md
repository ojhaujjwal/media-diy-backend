# Plan: EXIF schema strictness + indexed EXIF search

Follow-up to the completed dedup/s3key/exif implementation. Two work items chosen by the user:

1. **Schema strictness** — tighten `ExifMetadata` so `width`/`height` are required.
2. **Full EXIF search feature** — generated columns + indexes + repo query methods + RPC filters + tests.

Scope is the backend only (this repo). Frontend gallery/search UI lives in a separate repo and is out of scope.

---

## Background / current state (post-implementation)

- `ExifMetadata` (`src/domain/model/media.ts:20-45`): every field optional, so `exif: {}` is valid.
- `exif` persisted as a JSON `TEXT` column (`migrations/002_add_dedup_s3keys_exif.sql:11`). App does `JSON.parse`/`JSON.stringify`; never queries into the JSON from SQL.
- JSON field names in the blob: `make`, `model`, `gps.latitude`, `gps.longitude` (NOT `camera_make`/`gps_lat` — those are spec column names, not the JSON keys).
- `MediaMetadataRepository` (`src/domain/repository/media-metadata.repository.ts`): `create`, `findById`, `findByHash`, `findExistingSmbPaths`. No list/search method.
- RPC pattern for a query-style RPC: see `FindExistingMediaByFastScanRequest` (`src/http/rpc-handler/rpc-definitions.ts:58-70`) + handler + `tests/feature/find-existing-media-by-fast-scan.test.ts` (mock-repo handler tests — the codebase has no real-D1 test setup).
- `captured_at` is a top-level indexed-ready TEXT column (date-range queries need no EXIF extraction).

Verified facts (D1 docs + SQLite gencol docs):
- D1 supports `ALTER TABLE ADD COLUMN ... VIRTUAL` generated columns using `json_extract`.
- D1 allows indexes on generated columns (VIRTUAL or STORED).
- `STORED` columns cannot be added via ALTER TABLE; `VIRTUAL` can. We use VIRTUAL (instant, no backfill, indexable).
- SQLite indexes VIRTUAL generated columns fine (the "VIRTUAL can't be indexed" claim is MySQL-only).

## 1. Schema strictness — `src/domain/model/media.ts`

Change `ExifMetadata` `width` and `height` from optional to required:
```ts
width: S.Number,
height: S.Number,
```
All other fields (`make`, `model`, `gps`, exposure/aperture/etc.) stay optional.

Rationale: photos virtually always have dimensions. If the backup agent failed to extract dimensions, it should omit `exif` entirely (the field is `S.optional(ExifMetadata)` on `MediaMetadata`) rather than store a hollow object. Making `width`/`height` required also makes `exif: {}` invalid for free — no separate "non-empty struct" refinement needed (Effect Schema has no built-in for that, and it isn't required here).

Impact: the `UploadMediaRequest` payload already uses `exif: S.optional(ExifMetadata)`, so RPC validation enforces this at the boundary automatically. Existing tests don't send `exif`, so they're unaffected.

## 2. Migration `migrations/003_add_exif_generated_columns.sql`

Add VIRTUAL generated columns + indexes for the spec's queryable EXIF fields, extracting from the `exif` JSON blob using the actual JSON key names:
```sql
ALTER TABLE media_metadata
  ADD COLUMN camera_make TEXT GENERATED ALWAYS AS (json_extract(exif, '$.make')) VIRTUAL;
ALTER TABLE media_metadata
  ADD COLUMN camera_model TEXT GENERATED ALWAYS AS (json_extract(exif, '$.model')) VIRTUAL;
ALTER TABLE media_metadata
  ADD COLUMN gps_lat REAL GENERATED ALWAYS AS (json_extract(exif, '$.gps.latitude')) VIRTUAL;
ALTER TABLE media_metadata
  ADD COLUMN gps_lon REAL GENERATED ALWAYS AS (json_extract(exif, '$.gps.longitude')) VIRTUAL;

CREATE INDEX IF NOT EXISTS idx_media_camera_make  ON media_metadata(camera_make);
CREATE INDEX IF NOT EXISTS idx_media_camera_model ON media_metadata(camera_model);
CREATE INDEX IF NOT EXISTS idx_media_gps_lat      ON media_metadata(gps_lat);
CREATE INDEX IF NOT EXISTS idx_media_gps_lon      ON media_metadata(gps_lon);
```

Notes:
- VIRTUAL = computed on read, no storage, no backfill (instant add even on a populated table).
- When `exif` is NULL or the JSON key is absent, `json_extract` returns SQL NULL → generated column is NULL → row excluded from camera/GPS filters, included in unfiltered queries. Correct behavior.
- `json_extract` returns numeric values as REAL, so `gps_lat`/`gps_lon` as REAL supports range (bounding-box) queries.
- Generated columns are DB-internal for indexing. App code does NOT read them — `D1MediaRow` (`src/infrastructure/persistence/media-metadata.d1.ts:13-28`) is unchanged; `SELECT *` returns the generated columns but Effect `Schema.Struct` ignores unknown keys. The search handler derives `cameraMake`/`cameraModel` for the response from `media.exif?.make` / `media.exif?.model` (already parsed from the JSON blob), not from the generated columns.

## 3. Search feature — new `SearchMediaRequest` RPC

### 3a. Request/response schemas — new `src/http/request/search-media.request.ts`
```ts
export enum ERROR_CODE { SERVER_ERROR = "server_error" }
export class SearchMediaError extends S.TaggedErrorClass<SearchMediaError>()("SearchMediaError", {
  errorCode: S.Enum(ERROR_CODE)
}) {}

export class MediaSummary extends S.Class<MediaSummary>("MediaSummary")({
  id: S.String.check(S.isUUID()),
  s3KeyFull: S.String,
  s3KeyThumb: S.optional(S.String),
  type: S.Enum(MediaType),
  capturedAt: S.Date,
  cameraMake: S.optional(S.String),
  cameraModel: S.optional(S.String)
}) {}

export class SearchMediaResponse extends S.Class<SearchMediaResponse>("SearchMediaResponse")({
  results: S.Array(MediaSummary),
  total: S.Number
}) {}
```
Import `MediaType` from `src/domain/model/media.ts` (matches `FindMediaResponse` pattern).

### 3b. RPC definition — `src/http/rpc-handler/rpc-definitions.ts`
Add + register in `MediaRpcs`:
```ts
export const SearchMediaRequest = Rpc.make("SearchMediaRequest", {
  payload: {
    ownerUserId: S.String.check(S.isUUID()),
    dateFrom: S.optional(S.Date),
    dateTo: S.optional(S.Date),
    cameraMake: S.optional(S.String),
    cameraModel: S.optional(S.String),
    gpsLatMin: S.optional(S.Number),
    gpsLatMax: S.optional(S.Number),
    gpsLonMin: S.optional(S.Number),
    gpsLonMax: S.optional(S.Number),
    limit: S.Number,
    offset: S.Number
  },
  success: SearchMediaResponse,
  error: SearchMediaError
});
```
`ownerUserId` required (scopes by owner — matches existing `findById(ownerUserId, ...)` pattern; `family_id` still deferred). `limit`/`offset` required (matches codebase convention of no defaulted optional fields; caller passes explicit `50`/`0`).

### 3c. Repository method — `src/domain/repository/media-metadata.repository.ts`
Add to the `MediaMetadataRepository` service interface:
```ts
readonly searchMedia: (criteria: {
  readonly ownerUserId: string;
  readonly dateFrom?: Date;
  readonly dateTo?: Date;
  readonly cameraMake?: string;
  readonly cameraModel?: string;
  readonly gpsLatMin?: number;
  readonly gpsLatMax?: number;
  readonly gpsLonMin?: number;
  readonly gpsLonMax?: number;
  readonly limit: number;
  readonly offset: number;
}) => Effect.Effect<
  { readonly results: ReadonlyArray<MediaMetadata>; readonly total: number },
  MediaMetadataRepositoryError,
  never
>;
```

### 3d. D1 impl — `src/infrastructure/persistence/media-metadata.d1.ts`
Implement `searchMedia` with a dynamically-built WHERE clause shared by a COUNT query and a SELECT query:
- Always start `WHERE owner_user_id = ?`.
- Append `captured_at >= ?` / `captured_at <= ?` for `dateFrom`/`dateTo` (ISO strings).
- Append `camera_make = ?` / `camera_model = ?` (references the generated columns — sargable via their indexes).
- Append `gps_lat >= ?` / `gps_lat <= ?` / `gps_lon >= ?` / `gps_lon <= ?` for the GPS bounding box.
- Build `whereClause = where.join(" AND ")` and a `params` array.
- Run `SELECT COUNT(*) as c FROM media_metadata WHERE ${whereClause}` `.bind(...params).first<{c:number}>()` for `total`.
- Run `SELECT * FROM media_metadata WHERE ${whereClause} ORDER BY captured_at DESC LIMIT ? OFFSET ?` `.bind(...params, limit, offset).all<Record<string,unknown>>()` for rows.
- Map rows to `MediaMetadata` via a sync parse. Refactor: extract `parseRowSync(record: Record<string,unknown>): MediaMetadata` from the existing `parseRow` (which returns an Effect purely for the not-found branch) so list mapping is straightforward. `parseRow` keeps its Effect-returning signature for `findById`/`findByHash` (not-found handling) and delegates to `parseRowSync` for the happy path.
- Return `{ results, total }`.
- Use the existing `.pipe(Effect.mapError(mapD1Error), Effect.provideService(RuntimeContext, ctx))` pattern.

### 3e. Handler — new `src/http/rpc-handler/search-media.handler.ts`
Following the `find-existing-media-by-fast-scan.handler.ts` template:
- Destructure the payload fields.
- Call `repo.searchMedia(criteria)`.
- Map `results` to `MediaSummary[]`: `{ id, s3KeyFull, s3KeyThumb, type, capturedAt, cameraMake: media.exif?.make, cameraModel: media.exif?.model }`.
- Return `new SearchMediaResponse({ results: summaries, total })`.
- Use `errorHandler` from `helpers.ts` with `SearchMediaError/SERVER_ERROR`, plus `Effect.catchTags({ MediaMetadataRepositoryError: routeErrorHandler })` + `Effect.catchDefect(routeErrorHandler)`.

### 3f. Handler registration — `src/http/rpc-handler/media-rpc-handlers.ts`
Add `SearchMediaRequest: searchMediaHandler` to the `MediaRpcs.toLayer(...)` object + import.
