# Plan: Integration tests for D1 & R2 via alchemy `Test/Vitest` + infra cleanup

## Goal

Close the biggest test gap: the persistence layer (`MediaMetadataD1Live`, `MediaContentsR2Live`) and the real D1/R2 behavior are **entirely untested** — the 4 existing feature tests mock the repositories via `Layer.succeed` and never touch real D1/R2. Add **integration tests that exercise the real repository code against real Cloudflare D1 + R2**, preferring functional/behavioral coverage over implementation-detail unit tests. Also clean up the orphaned pre-migration infra (MinIO + DynamoDB Local + dead config) that no longer matches the D1/R2 app.

## Approach: real Cloudflare via alchemy `Test/Vitest`

The canonical alchemy pattern (`Test.make({ providers: Cloudflare.providers() })` + `beforeAll(deploy(Stack))`) deploys the **real app stack** to `stage: "test"` (auto-named ephemeral D1/R2/Worker) and drives it over **`RpcClient.make(MediaRpcs)` + `RpcClient.layerProtocolHttp({ url })`** — see `alchemy-effect` `packages/alchemy/test/Cloudflare/Workers/RpcHttp.test.ts`, `RpcWorker.test.ts`, `ChatPersistenceRpc.test.ts`. Migrations 001-003 auto-apply on deploy. `destroy(Stack)` in `afterAll` tears it down.

`MediaMetadataD1Live` / `MediaContentsR2Live` can only run inside the Worker runtime (`Cloudflare.D1Connection.bind(MediaDb)` requires `RuntimeContext` + `WorkerEnvironment` + `D1ConnectionLive`), so testing the real repo code ⇒ driving the deployed Worker over RPC. Direct D1 SQL assertions use `d1.queryDatabase` from `@distilled.cloud/cloudflare/d1` (helper at `alchemy-effect` `packages/alchemy/test/Cloudflare/D1/Database.test.ts:506`).

**Consequence:** testing D1/R2 integration through the deployed Worker *also* covers, for free, the 3 untested handlers (`findMediaById`, `findMediaByHash`, `generateUploadPresignedUrl`) and the full `RpcServer.toHttpEffect` + schema-validation HTTP boundary. Maximum functional coverage per test — aligned with the stated preference for functional tests over implementation details.

## Prerequisite code change (required for R2 test + fixes a latent prod bug)

`src/infrastructure/persistence/media-contents.r2.ts:12-24` reads `R2_ACCOUNT_ID`, `R2_BUCKET_NAME`, `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY` from `process.env` at **module load time**. In a deployed Cloudflare Worker, `process.env` only has values bound via the Worker resource's `env` (`alchemy/Cloudflare` `Worker.ts:272` — `Config.redacted`/`Config.string`/literals → `secret_text`/`plain_text`). The current `Worker` (`src/http/worker.ts:18-24`) binds **no** env/secrets, so `generatePresignedUrlForUpload` cannot work in production (empty creds → S3 errors).

Fix: bind R2 creds on the Worker so they reach `env`/`process.env` under `nodejs_compat`:
- `R2_ACCOUNT_ID`, `R2_BUCKET_NAME` as `Config.string`
- `R2_ACCESS_KEY_ID`, `R2_SECRET_ACCESS_KEY` as `Config.redacted` (secrets)

Resolve from `CloudflareEnvironment`/`Config` in the Worker init phase. `media-contents.r2.ts` keeps reading `process.env.R2_*` unchanged. This unblocks the R2 integration test and production presigned URLs.
